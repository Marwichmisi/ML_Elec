### Key Ideas Generated (capturées pendant l'exploration)

**[Category #1]**: Valeur Fantôme
_Concept_: Le produit échoue s'il ne résout pas une douleur métier immédiate et mesurable. La valeur perçue doit être visible rapidement pour les équipes industrielles.
_Novelty_: Cible la perception d'utilité métier avant toute sophistication technique.

**[Category #2]**: Performance Non Négociable
_Concept_: La fiabilité d'ingestion et la stabilité opérationnelle déterminent l'adoption. Sans performance robuste, l'outil est abandonné même si les fonctions sont riches.
_Novelty_: Place la performance comme condition d'existence produit.

**[Category #3]**: Feature-Market Misfit
_Concept_: Les fonctionnalités non prioritaires diluent l'impact et la lisibilité de la V1. Le produit doit livrer d'abord les capacités critiques terrain.
_Novelty_: Privilégie la hiérarchie de valeur plutôt que la quantité de fonctions.

**[Category #4]**: Time-to-First-Graph
_Concept_: L'onboarding idéal permet de connecter un microcontrôleur et tracer un premier graphe très vite. Ce premier succès conditionne l'adhésion utilisateur.
_Novelty_: Définit une métrique d'adoption immédiate concrète.

**[Category #5]**: Gouvernance Méritocratique
_Concept_: L'open source a besoin de règles explicites de contribution, revue et priorisation. La méritocratie crée confiance et durabilité communautaire.
_Novelty_: Traite la gouvernance comme fonctionnalité stratégique du produit.

## SCAMPER Method — Résultats (En Cours)

**Technique:** SCAMPER Method
**Interactive Focus:** Générer des variantes V1/plugins concrètes via 7 angles d'innovation appliqués au logiciel ML_Elec.

### S — Substitute (Substituer) — Idées Validées

**[Category #31]**: Protocole Universel Natif MQTT/OPC-UA
_Concept_: MQTT et OPC-UA comme couche d'ingestion native dès la V1 — le logiciel parle le langage des capteurs et automates industriels sans traduction ni adaptateur propriétaire. Connexion immédiate à tout l'écosystème matériel existant.
_Novelty_: Élimine la friction d'intégration hardware, transforme ML_Elec en hub universel de communication industrielle plutôt qu'outil isolé.

**[Category #32]**: Stockage Offline-First Souverain (Parquet/DuckDB)
_Concept_: Parquet/DuckDB comme moteur de stockage local — léger, offline-first, interrogeable en SQL standard, exportable sans dépendance externe. Les données restent sur la machine de l'utilisateur par design structurel.
_Novelty_: La souveraineté des données et la résilience edge deviennent des features produit différenciantes, pas de simples contraintes techniques.

**[Category #33]**: Architecture IA Bicouche (BYOM Prédiction + LLM Interprétation)
_Concept_: Deux couches IA à rôles distincts et séparés. Couche 1 — moteur de prédiction via BYOM (modèle ML importable, avec modèle de détection d'anomalies livré par défaut à l'installation). Couche 2 — moteur d'interprétation via LLM providers (OpenAI, Mistral, etc.) qui traduit les résultats ML en langage humain compréhensible. Le LLM peut ensuite répondre aux questions du technicien ou de l'ingénieur sur les résultats.
_Novelty_: Sépare explicitement "prédire" et "expliquer" — chaque couche est remplaçable indépendamment. Un technicien comprend le résultat sans être data scientist. Interface conversationnelle native sur les données industrielles.

**[Category #34]**: Webhook Universel comme Infrastructure de Notifications
_Concept_: Remplacer un système de notifications intégré complexe par un webhook universel — ML_Elec émet un payload JSON structuré, l'utilisateur branche son propre écosystème (Teams, Slack, n8n, Zapier, SMS, email, PagerDuty, etc.) sans maintenance interne côté produit.
_Novelty_: ML_Elec devient émetteur d'événements agnostique — zéro lock-in sur les canaux de communication, extensibilité infinie, maintenance quasi-nulle sur cette couche.

**[Category #35]**: Taxonomie Multi-Nature des Plugins
_Concept_: Le système de plugins couvre plusieurs natures fonctionnelles distinctes — Source (protocoles/capteurs), Visualisation (graphes/dashboards), ML (modèles de prédiction), Output (canaux de sortie), Action (déclenchement sur équipements), Règle Métier (logique de détection custom), Rapport (génération de documents), CMMS (intégration systèmes de maintenance), Audit (traçabilité/compliance), Edge Agent (agent léger déporté).
_Novelty_: L'écosystème de plugins devient un vrai système d'extension multi-dimensionnel, pas juste une liste de connecteurs.

### C — Combine (Fusionner) — Idées Validées

**[Category #37]**: Règle Métier comme Moteur de Labeling Actif
_Concept_: Une règle métier définie par le technicien (ex: "vibration > X = anomalie") génère automatiquement un dataset d'entraînement labellisé. Ces cas collectés alimentent l'affinage continu du modèle ML par défaut, sans intervention data science.
_Novelty_: Le technicien terrain devient acteur de l'amélioration du modèle ML sans le savoir — la règle est à la fois outil de détection ET outil de supervision d'entraînement.

**[Category #38]**: Dashboard Conversationnel (Chart Q&A via LLM)
_Concept_: Le dashboard devient interactif via le LLM — l'ingénieur clique sur un pic anormal d'une courbe et pose une question en langage naturel ("qu'est-ce qui s'est passé ici ?"). Le système extrait le contexte temporel autour du point (valeurs, annotations ML, métadonnées capteur) et l'envoie au LLM qui répond en langage humain.
_Novelty_: Transforme le dashboard de visualisation passive en outil de diagnostic conversationnel actif. Argument marché fort : "diagnostics industriels conversationnels" — aucun outil open source n'offre ça nativement.

**[Category #39]**: Synchronisation Sérialisée et Priorisée vers le Cloud
_Concept_: En mode offline, les données sont stockées localement en Parquet/DuckDB. Quand la connexion revient, la synchronisation est sérialisée et priorisée intelligemment — alertes critiques et anomalies détectées d'abord, données brutes ensuite, rapports en différé. Structure d'envoi optimisée pour le cloud récepteur.
_Novelty_: ML_Elec ne se contente pas de "travailler offline" — il gère la réconciliation des données avec intelligence de priorité métier, pas juste de l'ordre chronologique.

**[Category #40]**: Action Unifiée Règle+Plugin (Automatisation Traçable)
_Concept_: Plugin Action et Règle Métier fusionnés en un système d'action unifié — une règle peut déclencher directement un plugin action, mais avec log d'audit systématique, fenêtre d'annulation configurable, et validation technicien optionnelle selon le niveau de criticité défini.
_Novelty_: Réalise la promesse des guardrails "automatisation supervisée" — l'action est automatique mais toujours traçable, réversible et auditable. Deux modes : action immédiate (low-criticality) et action avec confirmation (high-criticality).

**V1 — Natures de plugins inclus dans le scope initial :**
- 🔌 Plugin Source (protocoles/capteurs)
- 📊 Plugin Visualisation (graphes/dashboards)
- 🧠 Plugin ML (modèles BYOM)
- 📤 Plugin Output (canaux de sortie / webhook)
- ⚙️ Plugin Action (actions supervisées sur équipements)
- 📋 Plugin Règle Métier (logique de détection custom, seuils, conditions multi-variables)

**V2+ — Natures de plugins reportées :**
- 📄 Plugin Rapport (génération PDF/HTML/Word/Excel)
- 🔗 Plugin CMMS (intégration SAP PM, IBM Maximo, eMaint)
- 📝 Plugin Audit / Compliance (traçabilité réglementaire)
- 🎭 Plugin Simulation / Replay (test modèles sur historique)
- 📡 Plugin Edge Agent (agent léger Raspberry Pi / PLC embarqué)

**[Category #36]**: Règle Métier comme Plugin V1
_Concept_: Logique de détection personnalisée définie par l'utilisateur via des règles configurables — seuils, conditions multi-variables, corrélations temporelles — sans nécessiter l'écriture d'un nouveau modèle ML.
_Novelty_: Démocratise la détection d'anomalies pour les techniciens non-data-scientists, complémentaire au BYOM pour les profils avancés.

### Décisions Architecture IA Validées (Issues du SCAMPER)

- **Modèle par défaut V1** : modèle de détection d'anomalies générique (type Isolation Forest ou Autoencoder) livré à l'installation, fonctionnel sans configuration.
- **BYOM** : l'utilisateur peut remplacer ou ajouter ses propres modèles ML (format ONNX ou Python).
- **LLM** : couche d'interprétation via API providers configurables — traduit les prédictions en langage naturel et supporte le Q&A contextuel sur les données.
- **Flux IA** : Données → Modèle ML (prédiction brute) → LLM (interprétation humaine + Q&A) → Technicien/Ingénieur.

### A — Adapt (Adapter) — Idées Validées

**[Category #41]**: Palette de Commandes Universelle (VS Code Style)
_Concept_: Un raccourci clavier universel (Ctrl+P ou équivalent) ouvre une palette de commandes permettant de piloter toutes les fonctions de ML_Elec sans quitter le clavier — "Connecter capteur", "Créer règle", "Lancer analyse", "Exporter rapport", "Changer modèle ML". Toutes les actions accessibles en 2 frappes.
_Novelty_: Transforme ML_Elec en outil pro à la hauteur des standards des ingénieurs — la vitesse d'exécution devient une feature UX différenciante, pas un détail.

**[Category #42]**: Versioning Git-Like des Modèles ML (Model Registry)
_Concept_: Chaque modèle ML importé via BYOM est versionné — historique de performance, métriques par version, possibilité de rollback instantané si les prédictions se dégradent. Le système surveille activement la dérive de performance et alerte si un modèle récent est moins précis que sa version précédente.
_Novelty_: Sécurise le BYOM en contexte industriel — le modèle peut évoluer ET se dégrader, le rollback est une garantie opérationnelle indispensable en production critique.

**[Category #43]**: Éditeur Visuel de Flux No-Code (n8n / Unreal Blueprint Style)
_Concept_: Configuration des workflows de maintenance via un éditeur visuel node-based — chaque plugin devient un nœud connecté visuellement : Capteur → Règle Métier → Modèle ML → Action → Output. Aucune connaissance du code requise. L'utilisateur assemble ses flux de surveillance par glisser-déposer, comme dans n8n ou les Blueprints d'Unreal Engine.
_Novelty_: Démocratise radicalement la configuration — un technicien terrain peut construire un flux de détection d'anomalie complet sans écrire une ligne de code. Les plugins deviennent des briques visuelles réutilisables et combinables à l'infini.

### M — Modify / Magnify (Modifier / Amplifier) — Idées Validées

**[Category #44]**: Onboarding 15 Minutes (KPI Officiel V1)
_Concept_: L'objectif d'onboarding est fixé à 15 minutes maximum — de l'installation à la première courbe visible avec détection d'anomalie active et LLM prêt. Zéro configuration manuelle requise pour ce premier succès. Ce KPI est une contrainte de design, pas une ambition vague.
_Novelty_: Transforme le Time-to-First-Value en critère de validation produit mesurable. Tout composant qui rallonge l'onboarding au-delà de 15 min est rejeté ou simplifié.

**[Category #45]**: A/B Testing ML en Shadow Mode (Manuel, User-Initiated)
_Concept_: Fonctionnalité opt-in déclenchée par l'utilisateur — le modèle actif prédit en production tandis qu'un modèle candidat prédit en shadow mode silencieux. ML_Elec compare les performances des deux et présente un tableau de bord de comparaison avant de proposer le basculement. Jamais activé automatiquement.
_Novelty_: Rend le déploiement de nouveaux modèles ML sûr et mesurable en contexte industriel — le technicien décide en connaissance de cause, pas à l'aveugle.

**[Category #46]**: Marketplace de Flux Visuels Communautaires
_Concept_: Les flux de surveillance construits dans l'éditeur visuel sont exportables et partageables comme templates — la communauté open source publie des flux prêts à l'emploi ("surveillance moteur électrique", "détection surchauffe transformateur", "monitoring onduleur solaire") importables en un clic dans n'importe quelle installation ML_Elec.
_Novelty_: L'éditeur visuel devient un hub de partage de savoir-faire industriel. Chaque flux partagé est une démonstration de valeur et un vecteur d'adoption — augmente la notoriété et la popularité du projet open source exponentiellement.

### P — Put to Other Uses (Nouveaux Usages) — Idées Validées

**[Category #47]**: ML_Elec comme Outil Pédagogique
_Concept_: Mode éducatif avec données de démonstration pré-chargées — les étudiants en écoles d'ingénieurs et lycées techniques connectent de vrais capteurs, visualisent des signaux réels, apprennent la maintenance prédictive par la pratique directe. Aucun développement supplémentaire requis, c'est le même core.
_Novelty_: Ouvre un second marché d'adoption naturel et gratuit. Les étudiants formés sur ML_Elec deviennent des prescripteurs en entreprise dès leur premier poste.

**[Category #48]**: ML_Elec comme Plateforme de Recherche Universitaire
_Concept_: Les laboratoires de recherche en génie électrique ont exactement les mêmes besoins que l'industrie — ingestion capteurs, visualisation, modèles ML custom via BYOM, mode offline sur équipements de labo. ML_Elec adopté sans modification par des laboratoires universitaires qui contribuent en retour des plugins de recherche avancés.
_Novelty_: Un labo qui cite ML_Elec dans une publication académique génère une visibilité et une crédibilité impossibles à acheter. Les chercheurs deviennent contributeurs de plugins de très haute valeur technique.

**[Category #49]**: ML_Elec pour l'Électricité Tout Domaine (Bâtiment, Énergie, Réseau)
_Concept_: La vision s'étend au-delà des moteurs industriels — ML_Elec peut surveiller tout système électrique et électrotechnique : tableaux électriques de bâtiments intelligents, onduleurs solaires, réseaux de distribution, équipements CVC, compteurs industriels. La prédiction s'applique à toute l'électricité.
_Novelty_: Repositionne ML_Elec non pas comme "outil de maintenance moteurs" mais comme "plateforme universelle de monitoring électrique et électrotechnique" — marché adressable multiplié par 10.

**[Category #50]**: ML_Elec comme Outil d'Audit Énergétique Terrain
_Concept_: Un auditeur énergétique connecte ML_Elec pendant 48h chez un client — collecte offline, analyse locale, puis génère automatiquement un rapport de consommation avec anomalies détectées. Les recommandations LLM sont produites lors de la reconnexion (LLM nécessite internet). Outil d'audit nomade complet sans infrastructure cloud permanente.
_Novelty_: Combine parfaitement offline-first + rapport généré + LLM interprétation différée. Un bureau d'études peut facturer des audits premium grâce à ML_Elec sans investissement infrastructure.

### Décision Technique LLM Offline (Validée)

- **V1** : Le LLM fonctionne uniquement avec connexion internet (API providers). En mode offline, ML_Elec opère en mode "prédiction seule" — le modèle ML local continue de détecter les anomalies, le LLM interprète lors de la reconnexion.
- **V2+** : Plugin LLM Local (Ollama, llama.cpp) pour interprétation IA complètement offline — même architecture, nouveau plugin.

### E — Eliminate (Éliminer) — Résultats

**[REFUSÉ] Éliminer la Configuration Initiale**
_Décision_: Rejeté par l'utilisateur. La configuration initiale est intentionnelle et nécessaire — elle garantit que l'utilisateur comprend ce qu'il configure et maintient la traçabilité des décisions de setup. Pas d'auto-magie opaque au démarrage.

**[Category #51]**: Écran d'Accueil "État de Santé" (Health Overview First)
_Concept_: L'écran principal de ML_Elec n'est pas un dashboard de courbes mais une vue "état de santé" synthétique — une ligne par équipement surveillé avec indicateur visuel vert/orange/rouge et motif d'alerte en clair. Le technicien sait "est-ce que ça va ?" en 2 secondes dès son arrivée, puis forage dans le détail si nécessaire.
_Novelty_: Inverse la priorité d'affichage classique — la synthèse d'abord, l'analyse ensuite. Correspond au workflow réel du technicien terrain, pas au workflow du data analyst.

**[Category #52]**: Architecture Pair-à-Pair Multi-Sites (P2P Mesh)
_Concept_: Plusieurs instances ML_Elec sur différents sites se synchronisent directement entre elles sans serveur central — chaque nœud est à la fois collecteur, analyseur et relais. Analyse en temps réel cross-sites, zéro point de défaillance unique (SPOF), déploiement multi-usines natif.
_Novelty_: Élimine la dépendance à une infrastructure centralisée tout en permettant des analyses corrélées entre sites. Un incident sur le site A peut déclencher une surveillance renforcée automatique sur le site B si les équipements sont similaires.

### R — Reverse / Rearrange (Inverser) — Résultats

**[Category #53]**: Ingestion Push Native (Context-Dépendant)
_Concept_: Selon les capteurs et les besoins, ML_Elec supporte nativement le mode push — l'équipement envoie activement ses données quand il détecte un événement inhabituel, sans attendre une interrogation périodique. Pull et push coexistent, la configuration du flux visuel détermine le mode.
_Novelty_: L'équipement devient acteur intelligent de sa propre surveillance, pas objet passif. Réduit la charge réseau en mode nominal et augmente la réactivité sur événements critiques.

**[Category #54]**: ML_Elec Interroge l'Ingénieur (Active Learning Terrain)
_Concept_: Quand ML_Elec détecte une anomalie ambiguë, il pose proactivement une question contextuelle à l'ingénieur ou au technicien : "J'ai détecté un pattern inhabituel sur le moteur M3 — est-ce qu'une maintenance était prévue hier ?" La réponse est capturée et utilisée pour affiner le modèle ML et enrichir le dataset de labeling.
_Novelty_: Le système apprend activement du savoir terrain implicite des experts humains. Le LLM formule la question, la réponse devient une donnée d'entraînement — boucle d'amélioration continue sans data scientist.

**[Category #55]**: Connecteurs Grafana / Home Assistant via Plugins (V2)
_Concept_: Plutôt qu'une stratégie d'adoption périphérique, ML_Elec propose en V2 des plugins de connexion bidirectionnelle avec Grafana, Home Assistant et autres outils existants — les utilisateurs de ces plateformes peuvent consommer les données et prédictions ML_Elec sans quitter leur environnement.
_Novelty_: Stratégie d'intégration écosystème plutôt que stratégie de remplacement — ML_Elec enrichit l'existant sans forcer une migration.

**[Category #56]**: Maintenance Prédictive Négative (Prédire Quand NE PAS Intervenir)
_Concept_: ML_Elec prédit non seulement les pannes mais aussi la durée de vie restante réelle des équipements — il alerte quand une maintenance planifiée est prématurée ou inutile. "Le roulement prévu pour remplacement vendredi montre 3 mois de vie résiduelle estimée — intervention non nécessaire."
_Novelty_: Inverse le paradigme classique de la maintenance prédictive. Réduit les coûts de maintenance inutile, démontre un ROI négatif de l'inaction (coût évité), argument commercial puissant auprès des directions industrielles.

### Dimension 3 — Contraintes Utilisateurs

| Élément | Décision | Impact |
|---------|----------|--------|
| Modes d'interaction | 🟢 Éditeur visuel ET CLI/Code — les deux en V1 | Double surface à maintenir |
| Confidentialité données LLM | 🟢 Adressé par Plugin LLM Local (V2 roadmap) — V1 : LLM désactivable | Clause de confidentialité claire dans la doc |
| Onboarding réseau | 🟢 Auto-détection appareils réseau local incluse en V1 | Critique pour KPI 15 minutes |
| Documentation | 🔴 Obligatoire V1 — explicite et complète | Bloquant release — pas de V1 sans doc |

**Implications et décisions issues de ces contraintes :**

- **Double mode V1** : l'éditeur visuel est le mode recommandé pour les techniciens, le CLI/code est le mode avancé pour les ingénieurs. Les deux accèdent aux mêmes fonctionnalités via des interfaces différentes.

- **Confidentialité LLM** : en V1, le LLM est optionnel et désactivable. Les utilisateurs en environnement sensible peuvent utiliser ML_Elec sans envoyer aucune donnée à l'extérieur. Le Plugin LLM Local (Ollama/llama.cpp) est la réponse V2 au besoin de confidentialité totale avec IA. Cette roadmap doit être explicitement documentée.

- **Auto-détection réseau local** : le wizard d'onboarding inclut un scan du réseau local pour détecter automatiquement les appareils MQTT et OPC-UA disponibles. L'utilisateur sélectionne, ne configure pas manuellement depuis zéro. Indispensable pour tenir le KPI 15 minutes.

- **Documentation V1 obligatoire** : la documentation est un livrable de sortie V1, pas une dette. Doit couvrir : installation, onboarding pas-à-pas, création de flux visuels, configuration plugins, référence API, FAQ. Rédigée de façon explicite pour les deux personas (technicien et ingénieur).

### Dimension 4 — Contraintes de Périmètre V1 (Scope)

| Élément | Décision V1 | V2+ |
|---------|------------|-----|
| Marketplace flux communautaires | 🟢 Export/Import fichiers locaux uniquement | Plateforme communautaire en ligne |
| Active Learning | 🟢 Capture de feedback utilisateur (V1) | Affinage automatique du modèle (V2) |
| Maintenance Prédictive Négative (RUL) | 🟢 Modèle RUL livré par défaut + BYOM pour remplacement | Modèles RUL spécialisés via marketplace |
| Multi-domaines | 🟢 UI/UX unique et générique — domaines adressés via plugins et flux | Configurations prédéfinies par domaine |

**Implications finales de scope :**

- **Marketplace V1** : uniquement export/import de fichiers `.mlflow` ou `.json` locaux. La plateforme de partage communautaire (type registry en ligne) est une infrastructure V2 distincte du core produit.

- **Active Learning V1** : quand ML_Elec détecte une anomalie ambiguë, il capture la réponse de l'ingénieur (oui/non/contexte) et la stocke en dataset labellisé. L'affinage automatique du modèle à partir de ce feedback est V2 — mais les données sont déjà collectées et prêtes.

- **Modèle RUL par défaut** : deux modèles livrés à l'installation — (1) Isolation Forest / Autoencoder pour la détection d'anomalies, (2) modèle RUL générique pour estimer la durée de vie résiduelle. Les deux remplaçables via BYOM.

- **UI/UX unique** : une seule interface qui s'adapte à tous les domaines via le système de plugins et les flux visuels. Pas de mode "bâtiment" ou mode "industrie" en dur — la généricité est une force, pas une faiblesse.

### Plan d'Action — Priorité 1 : Architecture Core & Data

**Pourquoi c'est fondamental :** Sans un core data solide, rien d'autre ne tient. C'est la fondation sur laquelle tout le reste est construit.

**Prochaines étapes immédiates :**

1. **Définir l'interface abstraite Plugin Source** — une interface commune pour MQTT et OPC-UA, extensible à d'autres protocoles via plugins futurs.
2. **PoC Parquet/DuckDB** — valider les performances d'ingestion, de requêtage et de synchronisation en conditions edge réelles.
3. **Concevoir la file de synchronisation priorisée** — 3 niveaux : alertes critiques (P1), données anomalies (P2), données brutes (P3).
4. **Spécifier l'API Core V1** — REST stable, versionnée `/v1/`, documentée OpenAPI dès le premier commit.
5. **Intégrer paho-mqtt + asyncua** — abstraits derrière l'interface Plugin Source.

**Ressources nécessaires :** Python, paho-mqtt, asyncua, DuckDB, pyarrow, FastAPI.
**Risque principal :** Performance OPC-UA en mode polling intensif — à benchmarker tôt.
**Indicateurs de succès :** Ingestion stable à 1000 points/seconde en local, synchronisation prioritaire validée en test offline/online.

---

### Plan d'Action — Priorité 2 : UX & Interfaces

**Pourquoi c'est fondamental :** Le KPI de 15 minutes est une contrainte de design qui gouverne chaque décision UX. Une interface mal pensée tue l'adoption même si le core est brillant.

**Prochaines étapes immédiates :**

1. **Wireframe Health Overview** — écran d'accueil : liste équipements + état vert/orange/rouge + motif d'alerte. Zéro clic pour voir l'état global.
2. **Concevoir le wizard d'onboarding 15 min** — étapes : installation → scan réseau local → sélection appareils → premier flux automatique → première courbe affichée.
3. **Implémenter l'auto-détection réseau local** — scan MQTT broker + OPC-UA servers sur le réseau local, présentation des appareils détectés avec noms lisibles.
4. **Concevoir la palette de commandes** — inventaire de toutes les actions ML_Elec, raccourci universel, recherche fuzzy.
5. **MVP Éditeur Visuel** — canvas avec nœuds glissables, 6 types de nœuds (un par nature de plugin), connexions visuelles, export/import JSON.
6. **Tests utilisateurs onboarding** — valider le KPI 15 min avec 3 profils : technicien junior, ingénieur senior, étudiant.

**Ressources nécessaires :** Framework frontend (React/Vue), librairie canvas (React Flow / Vue Flow), système de design cohérent.
**Risque principal :** L'éditeur visuel peut dépasser le scope si non encadré — livrer un MVP fonctionnel, pas un chef-d'œuvre.
**Indicateurs de succès :** 15 min d'onboarding validés par test utilisateur réel. Health Overview lisible en moins de 2 secondes.

---

### Plan d'Action — Priorité 3 : Valeur Métier & Marchés

**Pourquoi c'est fondamental :** La meilleure architecture et la meilleure UX ne valent rien si personne ne comprend pourquoi utiliser ML_Elec. La valeur doit être visible, immédiate et mesurable.

**Prochaines étapes immédiates :**

1. **Définir les 3 scénarios d'usage V1** — (a) maintenance industrielle moteurs, (b) monitoring bâtiment/tableau électrique, (c) audit énergétique terrain 48h. Chacun avec un flux visuel pré-construit livré dans la démo.
2. **Créer le mode démo** — données capteurs pré-chargées réalistes (moteur en bonne santé + moteur avec anomalie progressive), flux configuré, LLM prêt avec données demo. Onboarding éducatif possible sans aucun matériel.
3. **Formaliser le ROI framework** — calculateur simple : coût d'une panne évitée × nombre d'équipements × fréquence. À intégrer dans la documentation et le README.
4. **Identifier 3 early adopters** — un industriel, un bureau d'études énergie, un labo universitaire. Les impliquer dans le beta test V1 pour validation terrain.
5. **Rédiger le README "impact-first"** — présenter ML_Elec par la valeur métier (ce que ça évite, ce que ça prédit) avant les features techniques.

**Ressources nécessaires :** Données de démonstration réalistes, documentation claire, réseau de contacts early adopters.
**Risque principal :** Dilution du message en voulant adresser tous les domaines — maintenir un "hero use case" principal (maintenance industrielle) tout en restant générique.
**Indicateurs de succès :** Un early adopter peut décrire la valeur de ML_Elec en une phrase sans aide. README lu et compris par un technicien non-développeur.
